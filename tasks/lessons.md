# Lessons

- 2026-06-28: The model advisor should be chat-first and base-model-first. Hide existing MLX/CoreML/GGUF/quantized ports by default, reveal them only through an explicit toggle, and use autocomplete plus popular/recent base-model suggestions instead of a browsing dashboard.
- 2026-06-29: Advisor website copy should be compact and scannable. Prefer short labels such as route, family, and benchmark gate over explanatory sentences like "No portable speedup is confirmed until a local benchmark passes."
- 2026-06-29: Generated AI briefs must render as structured UI, not escaped prose. Split bullets, strip raw Markdown/backticks, and keep the brief compact.
- 2026-06-29: Do not make every source-backed route read like "needs local benchmark." Show observed working routes, known gaps, and source-reported outcomes separately; reserve benchmark language for numeric speed or memory claims.
- 2026-06-29: Model-advisor outcome cards must answer the upside question. Show an overall potential speedup range and speculative-decoding range, plus the gates that make the high end plausible; use a flat 1.0x-1.0x range when no credible speedup is known.
- 2026-06-30: MLX research maintenance should be delta-first. Keep a graph of already-read, indexed, new unread, and updated sources, then use the nightly delta to drive dynamic subagent sampling and separate review-only evidence from promoted skill/app/CLI changes.
- 2026-07-01: When the user asks to run the MLX porting skill through a model repo, default to a downloaded local model run and measured benchmark against the public/current MLX baseline before claiming skill quality or proposing README comparison copy.
- 2026-07-02: If the user corrects "Baseten and Wan2.2", treat the relevant comparison as Baseten's Wan2.2 optimization work, not Wav2Vec; carry over the lossless-vs-lossy and trained-gate framing only after checking it against the target model's measured bottleneck.
- 2026-07-02: For Amal-style technical blog drafts, use amaldavid.com for the evidence-first structure and amal.bearblog.dev for the sharper craft/validation voice. Preserve the technical caveats and receipts; do not turn the post into a generic benchmark announcement.
- 2026-07-06: Compound speedup estimates must be metric-scoped: never multiply a TTFT/prefill ratio into a decode-throughput ceiling, and never label a multiplied product with measured provenance. When a measured-together receipt exists, the measured ratio is the headline even when unfavorable: the dense stack measured 0.33x against the matched 8k baseline because the draft model regressed at long context, not because stacking is inherently bad. Measuring stacks caught that regression plus the mlx-lm 0.31.1 cache-composition limitation where target-only cache_prompt output does not compose with --draft-model.
- 2026-07-06: On Apple Silicon, measured optimization payoffs invert the paper-derived intuition: 4-bit weight quantization (2.4x decode) and prompt caching (23.8x TTFT on 8k prompts) dwarf speculative decoding (1.25x at k=2; k>=3 regresses). Recommend quant + prompt cache first; treat spec decode as a small, sweep-gated add-on.
- 2026-07-07: For Amal-style HN or launch copy, preserve the user's raw cadence, rough edges, and specific phrasing when explicitly requested. Do not normalize it into polished generic HN voice unless asked.
- 2026-07-07: For this HN narrative, do not frame agent models as unable to do ML optimization. The user's point is that models may be capable, but product/policy surfaces such as Fable's behavior and public government restrictions make these optimization workflows inaccessible or constrained; the skill exists to give agents a disciplined, auditable process around that gap.
- 2026-07-08: For MLX Porter public pages, avoid generic SaaS palettes; use MLX-specific metal, tensor, kernel, memory, parity, and checkpoint colors, keep GitHub prominent in the top nav, and verify mobile screenshots so the hero promise is not cropped.
- 2026-07-08: When the user says the MLX Porter site needs docs like a reference docs site, build a dedicated docs shell with sidebar navigation, search affordance, central article, code examples, next-step cards, and an on-page outline; do not treat the landing-page docs section as enough.
- 2026-07-08: For the MLX Porter hero background, do not put rectangular grid or frame lines behind oversized headline text; keep the softer nodes, rings, and connective lines so the hero feels technical without boxing the copy.
- 2026-07-08: For the MLX Porter landing hero, do not use terminal/process strips such as `agent: inspect -> oracle -> ...`; marketing-first means outcome cards that say what users can bring, what they get back, and what proof makes the result shippable.
- 2026-07-08: Do not add a color-palette/design-system section to the MLX Porter landing page. The product is CLI/runbook based, so the page should surface command paths, generated artifacts, and verification receipts instead of visual-token explanations.
- 2026-07-08: MLX Porter docs diagrams must be signal-heavy decision trees for developer/agent execution. Prefer route-specific Mermaid/open-canvas trees that map artifacts, scripts, runbooks, proof gates, and stop conditions over shallow summary cards.
- 2026-07-08: The MLX Porter landing page must not leave viewport-scale dead space between the docs band and final CTA. Keep the bottom transition compact enough that CTA copy and buttons are visible together on desktop and mobile screenshots.
- 2026-07-08: MLX Porter docs diagrams should behave like drill-down runbook mind maps for humans and agents. Use hover and double-click expansion, put 4-5 layers of technical depth where the repo has evidence, and attach numeric claims only to local receipts or clearly marked gates.
- 2026-07-08: For MLX Porter docs, do not turn decision diagrams into card boards or hidden expandable UI. Preserve a proper mind-map or connected diagram flow with all core branches visible; deepen the map itself rather than wrapping the content in cube/card interactions.
- 2026-07-08: MLX Porter docs mind maps should feel open and whimsical, not like dense machinery. Prefer sketchbook-style canvases, rounded note-like nodes, soft multi-color branches, and breathable branch spines over packed gray maps.
- 2026-07-08: When the user says whimsical mind maps, match the Whimsical-style visual language: pale open canvas, flowing curved branch strokes, mostly text-only nodes, and a small root anchor; avoid chunky colored boxes and visible grid-paper styling.
- 2026-07-09: MLX Porter landing hero copy should stay sparse. Prefer one sharp promise, one short support sentence, compact CTAs, and proof chips over stacked claims plus explanatory outcome cards.
- 2026-07-09: MLX Porter docs mind maps must preserve readability before whimsy. Route branch lines to the edge of labels, add a subtle label halo/background, and visually verify that text never sits directly on top of connector strokes.
- 2026-07-09: When decommissioning a competing MLX Porter runtime, remove its deployment and repository surface in the same release that promotes the static landing page and skill docs as canonical; do not leave two advisory products or orphan the public entry point.
- 2026-07-09: Historical speed receipts are performance observations, not optimization recommendations. Even a large measured ratio stays held until model lineage, checked-in workload hashes, raw aggregate recomputation, stability, baseline compatibility, output digests, and a task-quality artifact all pass; this supersedes earlier lessons that treated the July 6 ratios as recommendation-ready.
- 2026-07-10: When a full-project audit overlaps a conflicting nightly-curator PR, reconcile the immutable research receipts and current generated state into the audit branch, close the stale curator PR only after the replacement exists, and publish every authorized fix through one reviewable PR.
- 2026-07-12: The MLX Porter landing hero must teach the porting transformation, not merely display status bars. Use a compact mathematical motion loop—source semantics, weight/runtime translation, parity, then profile-gated optimization—with a complete static and reduced-motion equivalent.
